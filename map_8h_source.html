318 /**
319  * Manage and cache data about a part of the map.
320  *
321  * Despite the name, this class isn't actually responsible for managing the map as a whole. For that function,
322  * see \ref mapbuffer. Instead, this class loads a part of the mapbuffer into a cache, and adds certain temporary
323  * information such as lighting calculations to it.
324  *
325  * To understand the following descriptions better, you should also read \ref map_management
326  *
327  * The map coordinates always start at (0, 0) for the top-left and end at (map_width-1, map_height-1) for the bottom-right.
328  *
329  * The actual map data is stored in `submap` instances. These instances are managed by `mapbuffer`.
330  * References to the currently active submaps are stored in `map::grid`:
331  * 0 1 2
332  * 3 4 5
333  * 6 7 8
334  * In this example, the top-right submap would be at `grid[2]`.
335  *
336  * When the player moves between submaps, the whole map is shifted, so that if the player moves one submap to the right,
337  * (0, 0) now points to a tile one submap to the right from before
338  *
339  * The map's coverage is 11 * 11 submaps, corresponding to the current submap plus 5 submaps on every side of it,
340  * covering all Z levels. 5 submaps means 5 * 12 = 60 map squares, which means the map covers the current reality bubble
341  * in all directions for as long as you remain within the current submap (and when you leave the map shifts).
342  * The natural relative reference system for the map is the tripoint_bub_ms, as the map covers 132 * 132 tiles
343  * (11 * 11 submaps).
344  */
