32 // Some details about how achievements work
33 // ========================================
34 //
35 // Achievements are built on the stats_tracker, which is in turn built on the
36 // event_bus. Each of these layers involves subscription / callback style
37 // interfaces, so the code flow may now be obvious. Here's a quick outline of
38 // the execution flow to help clarify how it all fits together.
39 //
40 // * Various core game code paths generate events via the event bus.
41 // * The stats_tracker subscribes to the event bus, and receives these events.
42 // * Events contribute to event_multisets managed by the stats_tracker.
43 // * (In the docs, these event_multisets are described as "event streams").
44 // * (Optionally) event_transformations transform these event_multisets into
45 // other event_multisets based on json-defined transformation rules. These
46 // are also managed by stats_tracker.
47 // * event_statistics monitor these event_multisets and summarize them into
48 // single values. These are also managed by stats_tracker.
49 // * Each achievement requirement has a corresponding requirement_watcher which
50 // is alerted to statistic changes via the stats_tracker's watch interface.
51 // * Each requirement_watcher notifies its achievement_tracker (of which there
52 // is one per achievement) of the requirement's status on each change to a
53 // statistic.
54 // * The achievement_tracker keeps track of which requirements are currently
55 // satisfied and which are not.
56 // * When all the requirements are satisfied, the achievement_tracker tells the
57 // achievement_tracker (only one of these exists per game).
58 // * The achievement_tracker calls the achievement_attained_callback it was
59 // given at construction time. This hooks into the actual game logic (e.g.
60 // telling the player they just got an achievement).

793 {
