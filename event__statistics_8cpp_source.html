30 // event_transformation and event_statistic are both objects defined in json
31 // and managed via generic_factory.
32 // They both use the pimpl idiom and dynamic polymorphism to support their
33 // implementation.
34 // Their values can be calculated directly, or watched for updates.
35 // To register something that watches for updates, call
36 // stats_tracker::add_watcher.
37 //
38 // The way watching works is as follows:
39 // - Each event_transformation and event_statistic has a corresponding state
40 // object, which can be stored in the stats_tracker to hold the current
41 // state of that transformation or statistic.
42 // - Also, the stats_tracker stores all the watchers; any object watching any
43 // event type or the value of any transformation or statistic.
44 // - Each stat object will also register itself as a watcher for any event
45 // type, transformation, or statistic it depends on.
46 // - When an event occurs, watchers of that event_type will be informed.
47 // - When stats are thus informed, they update their value if appropriate, and
48 // inform the stats_tracker of the new value.
49 // - The stats_tracker then notifies watchers again in turn.
50 //
51 // In this way, the updates cascade to all relevant objects and ultimately the
52 // end-users (also in the form of watchers) are informed. For example, an
53 // achievement might watch in this way so that it can trigger itself when its
54 // conditions become true.
