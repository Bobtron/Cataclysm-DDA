180 /**
181  * The *_vector members represent list of alternatives requirements:
182  * alter_tool_comp_vector = { * { { a, b }, { c, d } }
183  * That means: the player needs (requirement a or b) and (requirement c or d).
184  *
185  * Some functions in this struct use template arguments so they can be
186  * applied to all three *_vector members. For example:
187  * check_consistency iterates over all entries in the supplied vector and calls
188  * check_consistency on each entry.
189  * If called as <code>check_consistency(tools)</code> this will actually call
190  * tool_comp::check_consistency. If called as
191  * <code>check_consistency(qualities)</code> it will call
192  * quality_requirement:check_consistency.
193  *
194  * Requirements (item_comp, tool_comp, quality_requirement) must have those
195  * functions:
196  * Load from an entry of a json array:
197  * void load(const JsonValue &value);
198  * Check whether the player has fulfills the requirement with this crafting
199  * inventory (or by mutation):
200  * bool has(const inventory &crafting_inv) const;
201  * A textual representation of the requirement:
202  * std::string to_string() const;
203  * Consistency check:
204  * void check_consistency(const std::string &display_name) const;
205  * Color to be used for displaying the requirement. has_one is true of the
206  * player fulfills an alternative requirement:
207  * nc_color get_color(bool has_one, const inventory &crafting_inv) const;

432 // Sometimes the requirement_data is problematic, because it has overlapping
433 // requirements. For example, a survivor telescope requires
434 // 1 high-quality lens
435 // 1 high-quality lens OR 1 small high-quality lens
436 // If there is just one high-quality lens in the available inventory then it's
437 // hard to correctly detect that these requirements are impossible to satisfy.
438 // In general, determining craftability is equivalent to boolean
439 // satisfiability, and thus NP-hard.
440 //
441 // In practice, we don't expect recipes to have too much overlap, so this issue
442 // should be tractable.
443 //
444 // However, to avoid keeping additional state during the process of searching
445 // for components, we don't make the component search more complex, instead we
446 // make the requirements more complex. We replace each requirement_data with a
447 // deduped_requirement_data, which contains a selection of alternative
448 // requirement_data objects, each of which contains no overlapping
449 // requirements.
450 //
451 // For the majority of recipes, there are no overlaps, and this will be
452 // essentially equivalent to just a requirement_data. However, for the few
453 // problematic recipes this allows us to calculate crafting requirements more
454 // accurately.
