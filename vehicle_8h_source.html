729 /**
730  * A vehicle as a whole with all its components.
731  *
732  * This object can occupy multiple tiles, the objects actually visible
733  * on the map are of type `vehicle_part`.
734  *
735  * Facts you need to know about implementation:
736  * - Vehicles belong to map. There's `std::vector<vehicle>`
737  * for each submap in grid. When requesting a reference
738  * to vehicle, keep in mind it can be invalidated
739  * by functions such as `map::displace_vehicle()`.
740  * - To check if there's any vehicle at given map tile,
741  * call `map::veh_at()`, and check vehicle type (`veh_null`
742  * means there's no vehicle there).
743  * - Vehicle consists of parts (represented by vector). Parts have some
744  * constant info: see veh_type.h, `vpart_info` structure that is accessible
745  * through `vehicle_part::info()` method.
746  * The second part is variable info, see `vehicle_part` structure.
747  * - Parts are mounted at some point relative to vehicle position (or starting part)
748  * (`0, 0` in mount coordinates). There can be more than one part at
749  * given mount coordinates, and they are mounted in different slots.
750  * Check tileray.h file to see a picture of coordinate axes.
751  * - Vehicle can be rotated to arbitrary degree. This means that
752  * mount coordinates are rotated to match vehicle's face direction before
753  * their actual positions are known. For optimization purposes
754  * mount coordinates are precalculated for current vehicle face direction
755  * and stored in `precalc[0]`. `precalc[1]` stores mount coordinates for
756  * next move (vehicle can move and turn). Method `map::displace_vehicle()`
757  * assigns `precalc[1]` to `precalc[0]`. At any time (except
758  * `map::vehmove()` innermost cycle) you can get actual part coordinates
759  * relative to vehicle's position by reading `precalc[0]`.
760  * Vehicles rotate around a (possibly changing) pivot point, and
761  * the precalculated coordinates always put the pivot point at (0,0).
762  * - Vehicle keeps track of 3 directions:
763  * Direction | Meaning
764  * --------- | -------
765  * face | where it's facing currently
766  * move | where it's moving, it's different from face if it's skidding
767  * turn_dir | where it will turn at next move, if it won't stop due to collision
768  * - Some methods take `part` or `p` parameter. This is the index of a part in
769  * the parts list.
770  * - Driver doesn't know what vehicle he drives.
771  * There's only player::in_vehicle flag which
772  * indicates that he is inside vehicle. To figure
773  * out what, you need to ask a map if there's a vehicle
774  * at driver/passenger position.
775  * - To keep info consistent, always use
776  * `map::board_vehicle()` and `map::unboard_vehicle()` for
777  * boarding/unboarding player.
778  * - To add new predesigned vehicle, add an entry to data/raw/vehicles.json
779  * similar to the existing ones. Keep in mind, that positive x coordinate points
780  * forwards, negative x is back, positive y is to the right, and
781  * negative y to the left:
782  *
783  * orthogonal dir left (-Y)
784  * ^
785  * -X -------> +X (forward)
786  * v
787  * orthogonal dir right (+Y)
788  *
789  * When adding parts, function checks possibility to install part at given
790  * coordinates. If it shows debug messages that it can't add parts, when you start
791  * the game, you did something wrong.
792  * There are a few rules:
793  * 1. Every mount point (tile) must begin with a part in the 'structure'
794  * location, usually a frame.
795  * 2. No part can stack with itself.
796  * 3. No part can stack with another part in the same location, unless that
797  * part is so small as to have no particular location (such as headlights).
798  * If you can't understand why installation fails, try to assemble your
799  * vehicle in game first.
800  */
